## What the PDF asks for

The two-page prompt distils into the following concrete requirements:

**Product requirements**
1. Manage the full lifecycle of a time-off request.
2. Maintain balance integrity between ExampleHR and an HCM system (Workday/SAP class), where HCM is the source of truth.
3. Handle cases where HCM balance changes independently (work-anniversary credits, year-start resets).
4. HCM provides a **realtime API** for getting/sending individual time-off values (per employee, per location).
5. HCM provides a **batch endpoint** that pushes the whole corpus of balances to ExampleHR.
6. Be **defensive** against unreliable HCM error reporting — HCM may silently accept invalid requests.
7. Serve two user personas: Employee (accurate balance + instant feedback) and Manager (approve/reject with confidence data is valid).

**Deliverables**
- A well-written Technical Requirements Document (TRD) with challenges, suggested solution, and analysis of alternatives.
- Code in a GitHub repository.
- A rigorous test suite as the primary agentic-development artifact.

**Technical guides**
- NestJS + SQLite.
- Real mock HCM server (not just stubs) as part of the test suite.
- Balances are per-employee per-location.
- Agentic development: no manual code written — TRD and test rigor are the deliverable.

## Bonus / above-spec features

The implementation delivers several features that go beyond what the PDF requires:

| Feature | Where |
|---|---|
| **Employee management (full CRUD)** | `apps/timeoff-api/src/modules/employees/` — `EmployeesController` (GET/POST/PATCH/DELETE), `EmployeeRepository`, `EmployeeSeederService` (auto-seeds Alice/Mary/Eddie/Emma on first boot) |
| **Real role-based access control (RBAC)** | `RolesGuard` + `@Roles()` decorator; ownership checks (employee sees own data, manager sees direct-reports, admin sees all); `RequireAuth` guard in frontend with `roles` array |
| **Frontend SPA** | `apps/frontend/` — Vite + React + Tailwind; Login page (employee directory), Employee page (balance + request), Manager page (approve/reject), Admin page (employee CRUD, HCM seeding, force-fail, quick demo setup) |
| **Manager team-aware filtering** | Requests list filtered by `managerId` relationship; balance reads scoped to direct reports |
| **Force-fail recovery endpoint** | `POST /requests/:id/force-fail` (admin) — manual recovery for DLQ'd sagas; releases reservation and sets `FAILED` |
| **Transactional outbox** | `OutboxEntry` table; dispatcher polls and publishes to BullMQ atomically; guarantees no dual-write hazard if worker crashes |
| **Idempotency on request creation** | Unique index on `idempotencyKey`; re-sending returns existing request transparently (T-4 proof test: 5 concurrent same key → exactly 1 request created) |
| **Structured observability** | Pino JSON logs with `requestId`, `sagaId`, `employeeId`, `correlationId` propagated via async-local-storage; `DRIFT_DETECTED` log when HCM delta > 5 days |
| **Liveness + readiness endpoints** | `GET /health` and `GET /health/ready` (probes DB, Redis, HCM); returns structured report |
| **Swagger / OpenAPI** | `GET /docs` served via `@nestjs/swagger` |
| **Docker-compose one-liner** | 5 services: api, worker, redis, hcm-mock, frontend (nginx); `docker compose up --build` is all that is needed |
| **ErrorBoundary in frontend** | React `ErrorBoundary` component wraps all routes; prevents white-screen crashes |
| **Property-based testing** | `fast-check` generates random action sequences and asserts `available >= 0` invariant (caught a real bug during development: when HCM pushed total below pending reservations, available went negative) |
| **Optimistic locking** | `version` column on `Balance`; every update uses `WHERE version = ?` to detect concurrent writers; caller retries on mismatch |

## Overall verdict

The implementation is **substantially compliant** with all PDF requirements. The core system — request lifecycle, HCM sync (realtime + batch), defensive validation, movement ledger, saga state machine, transactional outbox, BullMQ retry + DLQ, NestJS + SQLite, real HCM mock server, and test rigor — is fully built and all 79 tests pass.

The gaps (Prometheus metrics, overlap validation, per-module coverage thresholds, `?strategy=fresh`) are explicitly descoped in TRD §12, not overlooked. The implementation substantially exceeds the PDF in areas not asked for: Employee CRUD, RBAC, frontend SPA, and observability tooling.
